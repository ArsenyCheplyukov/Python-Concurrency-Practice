# 01 — Threading

Эксперименты с `threading` модулем: race conditions, GIL и Lock.

## Контекст

- Python 3.14.4, standard build (GIL enabled).
- В Python 3.10+ интерпретатор стал реже переключать GIL в "tight loop" из примитивов — нужны eval breakers (вызовы функций) внутри цикла, чтобы race condition воспроизвелась стабильно.
- Все примеры используют `sys.setswitchinterval(0.000001)` для максимизации шанса переключения.

## Файлы

- `01_race_condition_without_lock.py` — счётчик без Lock, 4 потока × 100k инкрементов. Результат **меньше 400_000**, разный каждый запуск. Демонстрация lost updates.
- `02_lock_fix.py` — тот же счётчик с `threading.Lock` вокруг инкремента. Результат всегда **400_000**. Цена корректности: ~2× медленнее.
- `03_bank_transfer.py` — два счёта, 3 потока переводят деньги. Три версии:
  - без Lock → сумма поплывёт
  - spot lock (узкая критсекция) → корректно, дороже
  - wide lock (на весь цикл) → корректно, дешевле, но полностью сериализует потоки

## Ключевые выводы

1. **`counter += 1` не атомарно** — это три байткод-инструкции (LOAD, BINARY_OP, STORE), между которыми может вклиниться другой поток.
2. **Lock = единственная гарантия**. Поведение GIL — implementation detail и меняется от версии к версии.
3. **Lock granularity** — критсекция должна покрывать ровно то, что обязано быть атомарным. Слишком широкий lock убивает параллелизм, слишком узкий — оставляет race condition.
4. **TOCTOU** — проверка инварианта и действие по нему должны быть в одной критсекции, иначе условие устареет.
